# Процесс разработки по спецификациям

Правила для проекта «Игра Azul против бота» (и любых следующих фич). Главный принцип: **ни одно изменение кода не делается без спецификации**. Спека — единственный источник требований для агента-исполнителя и единственный критерий приёмки для агента-ревьюера.

## Роли

| Роль            | Кто                    | Делает                                                                                                                    |
| --------------- | ---------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **Оркестратор** | Основная сессия Claude | Пишет спеки, утверждает с владельцем, запускает субагентов, интегрирует результат, коммитит, деплоит                      |
| **Исполнитель** | Субагент               | Реализует ровно одну спеку: код + все тесты из тест-плана. На вход получает спеку и контракты зависимостей, ничего больше |
| **Ревьюер**     | Отдельный субагент     | Получает спеку и дифф. Проверяет каждый FR по чек-листу, ищет нарушения инвариантов и выход за scope. Вердикт по пунктам  |
| **Владелец**    | @petrochen             | Утверждает спеки (или делегирует мелкие оркестратору), принимает результат на живом предпросмотре                         |

## Жизненный цикл спеки

```
draft → approved → implemented (+ amendments)
```

1. Оркестратор пишет спеку в `specs/NNN-имя.md` со статусом `draft`.
2. Владелец утверждает → `approved`. Мелкие технические спеки оркестратор может утверждать сам, отметив это в шапке.
3. Агент-исполнитель реализует. Возвращает: изменённые файлы, результат прогона тестов, самоотчёт-чеклист по каждому FR.
4. Агент-ревьюер сверяет дифф со спекой. Замечания → исполнителю на доработку (или оркестратор чинит сам, если тривиально).
5. Оркестратор прогоняет тесты локально, коммитит с сообщением `SPEC-NNN: <суть>`, ставит в шапке спеки статус `implemented` и хеш коммита.
6. **Багфиксы после приёмки** — только через поправку: в конец спеки добавляется раздел `## Amendments` с датой, описанием и причиной. Код и спека не имеют права расходиться.

## Требования к составу спецификации

Каждая спека обязана содержать все разделы (пустой раздел — явное «нет», а не пропуск):

1. **Шапка**: ID (`SPEC-NNN`), название, статус, зависимости (`SPEC-NNN`), **whitelist затрагиваемых файлов** — исполнитель не имеет права менять файлы вне списка.
2. **Цель** — один абзац: зачем это существует, для кого.
3. **Не-цели** — что сознательно вне scope. Защита от расползания: всё, что не в FR и попадает под не-цели, исполнитель не делает, даже если «удобно заодно».
4. **Контракты** — точные публичные интерфейсы: сигнатуры функций с типами аргументов и возврата, схемы структур данных (пример JSON), форматы сообщений Web Worker. Контракт — закон: его изменение возможно только поправкой к спеке, от которой зависят другие.
5. **Функциональные требования** — нумерованный список `FR-1…FR-N`. Каждое — проверяемое утверждение с конкретикой («`legalMoves` не возвращает ходов в линию, где лежит другой цвет»), никаких «должно быть удобно».
6. **Инварианты** — что не может нарушаться ни в каком состоянии (например: «сумма плиток по всем зонам всегда 100», «счёт ≥ 0»). Ревьюер проверяет их отдельно от FR.
7. **Критерии приёмки** — таблица: FR → способ проверки (юнит-тест / сценарий Playwright / прогон стенда / ручной чек на предпросмотре) с конкретными числами и порогами.
8. **Тест-план** — список тестов, которые исполнитель обязан написать, включая негативные случаи. Тесты — часть поставки, не опция.
9. **Бюджеты** — производительность, память, размер бандла, если применимо (например: «ход бота ≤ 1.5 с на M-серии»).
10. **Запрещено** — явный список: какие файлы не трогать, какое поведение не менять (например: «не менять `site/index.html`», «не добавлять внешние зависимости»).

## Модели субагентов (политика утверждена владельцем 2026-07-22)

Гибрид: **Fable** — только для критичных по корректности спек (SPEC-001 движок, SPEC-004 бот L2). **Sonnet** — исполнители SPEC-003/005/006, все ревьюеры и прочие механические задачи по подробной спеке.

## Конвенции кода и репозитория

- Язык: чистый JavaScript (ES-модули), без сборщиков и внешних runtime-зависимостей. Тесты — `node:test` (`node --test`).
- Движок и бот — чистые функции над plain-JSON состоянием; вся случайность — через seed-нный ГПСЧ, передаваемый явно (воспроизводимость партий обязательна).
- Коммит на спеку: `SPEC-NNN: <суть>`. Одна спека — один коммит (плюс отдельные коммиты поправок).
- UI-спеки принимаются на превью-деплое: `wrangler pages deploy site --project-name azul-score --branch spec-NNN` даёт отдельный URL, прод не затрагивается. В прод (`--branch main`) деплоит только оркестратор после приёмки владельцем.
- Секретов в репозитории нет. Ключ Nano Banana — только из macOS Keychain (`security find-generic-password -s "gemini-api-key" -w`), генерация ассетов — только локально на этапе сборки, сгенерированные PNG коммитятся.
